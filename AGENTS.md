# u校集 Codex 工作规则

## 项目说明

- 项目名：u校集
- 项目类型：校园二手闲置 + 求购微信小程序
- 技术栈：微信小程序、WXML、WXSS、JS、微信云开发、云函数、云数据库

## 默认工作原则

- 优先稳定，不要大规模重构。
- 默认只做最小 diff 修改。
- 不要随意修改云函数、数据库结构、`app.json`、`project.config.json`。
- 不要删除已有业务逻辑。
- 不要一次性重写多个核心文件。
- 修改前先说明会改哪些文件。
- 修改后只输出：改了哪些文件、如何测试、是否有风险。
- 回复尽量省 token，只说关键结论。

## UI 风格

u校集整体风格是：

- 神秘校园集市
- 校园隐藏商店
- RPG 商店感
- 悬赏告示栏
- 木牌
- 牛皮纸
- 暖灯
- 暗绿
- 琥珀黄
- 暗橘
- 神秘但安全
- 有游戏感但不阴暗

不要做成：

- 普通电商风
- 企业后台风
- 过度可爱卡通
- 纯白极简风
- 恐怖风
- 违法感
- 脏乱感
- 地下交易感

对外文案不要出现：

- 黑市
- 后街
- 地下交易
- 暗巷

## 首页核心文案

保留这些方向：

- u校集
- 集市告示
- 集市快报
- 集市秘闻
- 集市规约
- 淘货街区
- 出物奇摊
- 求购悬赏

不要使用“筛选”这个词。

## 发布逻辑

- 发布页包含两种 `publishType`：
  - `idle` = 我想卖
  - `wanted` = 我想收
- 两者共用 `goods` 集合，通过 `publishType` 区分。
- 不要随意拆分 `wanted` 集合。
- 不要破坏现有发布、详情、私信、我的发布流程。

## 图片展示规则

商品列表：

- 首页可以使用瀑布流。
- 图片完整展示。
- 不用 `aspectFill` 裁切。
- 不用固定高度 + `aspectFit` 留白。

我的发布页面：

- 保持紧凑管理列表。
- 不要套用首页大卡片/瀑布流样式。
- 不要把商品展示框做得过大。

商品详情：

- 多图可保留 `swiper`。
- 图片不要裁切。
- 保留 `wx.previewImage`。

## WXML 修改规则

修改 WXML 后必须检查：

- 所有标签是否闭合。
- 所有 `<text>` 是否有 `</text>`。
- 是否有错误闭合，如 `?/text>`。
- 是否有中文乱码，如 `鍔犺浇`、`涓婃粦`、`宸茬粡`。
- `wx:if` / `wx:elif` / `wx:else` 顺序是否正确。
- 文件保持 UTF-8 编码。
- 微信开发者工具能否编译。

## Preview 规则

如果是 UI 视觉探索，优先只改：

- `preview/index.html`
- `preview/style.css`
- `preview/assets/*`

不要直接改 `miniprogram`。等 preview 满意后，再迁移到小程序。

## 安全规则

- 不要提交密钥。
- 不要写入 AppSecret。
- 不要在小程序前端放 AI API Key。
- 不要绕过权限校验。
- 不要直接暴露 `messages`、`school_auth_requests` 等敏感集合。
- 涉及用户隐私、学校认证、私信、举报、AI 接口时必须谨慎说明风险。

## Figma / MCP 规则

使用 Figma MCP 前必须先做最小连接测试：

- 先读取文件。
- 再创建并删除 MCP test 文本节点。
- 成功后才执行正式写入。
- 失败只返回错误原文，不反复重试。
- 大任务拆成小批次执行。
- Figma 不稳定时改做 preview 网页版。

## 完成后输出格式

每次完成任务后只输出：

- 改了哪些文件
- 每个文件改了什么
- 如何测试
- 是否有风险
